# 🤝 Contributing to GitScore

Thank you for your interest in contributing to the GitScore coordinates! We welcome all explorers, from those fixing typos in the scrolls to those implementing deep scrying algorithms.

## 🏰 Our Standards

We strive for:
- **Visual Excellence**: All UI changes must maintain the mystical, premium aesthetic (glassmorphism, vibrant gradients, smooth animations).
- **Performance First**: New components should be lazy-loaded if heavy, and API calls should be optimized for edge delivery.
- **TypeScript Strictness**: No implicit `any`. Use proper interface definitions for all GitHub data structures.

## 🛠 Development Workflow

1. **Fork & Branch**: Create a feature branch from `main`.
2. **Setup**: Follow the [README.md](./README.md) instructions to get your local environment running.
3. **Drafting**: Use `npm run lint` and `npm run build` to verify your changes don't break the build rituals.
4. **Committing**: We prefer descriptive, prefix-based commit messages:
   - `feat:` for new mystical features
   - `fix:` for fixing rifts in the logic
   - `perf:` for speed optimizations
   - `docs:` for scroll updates

## 🧪 Testing Rituals

- **Unit Tests**: Ensure scoring logic in `lib/scoring` is tested for various edge cases (new users, empty repos, high-activity veterans).
- **Visual Testing**: Verify UI consistency across Dark/Mystical mode.

## 🗳 Submission Process

1. Open a Pull Request with a clear description of the transmutation.
2. Include screenshots or loom videos for any UI changes.
3. Ensure the Vercel Preview builds successfully.
